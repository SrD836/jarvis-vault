# Code Audit — 2026-05-30

> Auditoria PARALELA autonoma (`hermes/overnight_audit.py`, DeepSeek). FASE 5. Clusters auditados: observability, llm-routing, memory, vault-maintenance, autonomous-loops. Fallidos: ninguno.

## Resumen ejecutivo
Se identificaron 15 hallazgos en 5 clusters auditados, con 3 hallazgos recurrentes de auditorías previas que persisten sin corrección. Los problemas críticos incluyen manejo inadecuado de excepciones, dependencias frágiles en consultas SQL, y configuraciones hardcodeadas que comprometen la portabilidad del sistema. Se requiere acción inmediata en los hallazgos de severidad ALTA.

## Hallazgos por severidad
**ALTA**
- `run_logger.py`: Captura genérica de `Exception` enmascara errores reales del subproceso (rc negativo por señales)
- `llm_call.py`: No maneja `URLError` ni `JSONDecodeError`; `_fmt_err` puede lanzar `AttributeError` no capturado
- `memory_sqlite.py`: `_fts_query()` no escapa comillas dobles en tokens FTS5 → `OperationalError`
- `clean_trash.py`: Paths hardcodeados `/home/agent/` (recurrente)
- `learnings.py`: `deepseek_propose` pasa `env` a `llm_call` que no acepta el parámetro → autenticación rota

**MEDIA**
- `run_logger.py`: Rotación sobrescribe backups sin compresión ni retención
- `runs_dashboard.py`: `parse_cron_runs` asume timestamp numérico pero recibe string ISO → `TypeError`
- `llm_call.py`: `_notify_once` depende de `run_logger._utc_now()` sin fallback
- `memory_sqlite.py`: Deduplicación con `LIKE` frágil vs `json_extract`
- `memory_lib.py`: Captura genérica de `Exception` en dual-write oculta errores de DB

**BAJA**
- `runs_dashboard.py`: `parse_vault_runs` no filtra archivos por fecha real
- `llm_call.py`: Falta timeout en llamadas a contenedores Docker
- `memory_lib.py`: `now_iso()` naive sin timezone inconsistente con `memory_sqlite`
- `memory_backfill.py`: `parse_table()` no maneja filas vacías intermedias

## Hallazgos por cluster (crudo)

### observability
`run_logger.py` · ALTA · `_run_wrapper` captura `Exception` genérica, no `FileNotFoundError`/`PermissionError` específicas, y enmascara errores reales del subproceso como rc=127. Si el comando falla por señal (ej. SIGKILL), `subprocess.run` lanza `CalledProcessError` que NO es `Exception`? Sí lo es (hereda de `Exception`), pero el bloque captura todo, incluyendo `KeyboardInterrupt` (hereda de `BaseException` no de `Exception`) — correcto. Sin embargo, si el subproceso es terminado por señal, `subprocess.run` no lanza excepción, devuelve `returncode` negativo. El código no maneja returncode negativo explícitamente, lo trata como "error" genérico. No es bug crítico, pero la captura genérica de `Exception` es demasiado amplia y puede ocultar bugs del wrapper.

`run_logger.py` · MEDIA · `_maybe_rotate` rota a `path + ".1"` sobrescribiendo cualquier backup anterior sin compresión ni retención. En producción con rotaciones frecuentes (50MB), se pierde el historial completo. Debería rotar con timestamp o mantener N backups.

`runs_dashboard.py` · MEDIA · `parse_cron_runs` asume que `rec.get("ts")` es timestamp en milisegundos (lo divide por 1000), pero `run_logger.py` guarda `ts` como string ISO 8601 (`_utc_now()`), no como número. Esto causa `TypeError` en `dt.datetime.fromtimestamp(ts_ms / 1000, TZ)` cuando `ts_ms` es string. El dashboard nunca mostrará runs de cron.

`runs_dashboard.py` · BAJA · `parse_vault_runs` lee todos los archivos `.md` del directorio de hoy sin filtrar por fecha/hora real del archivo. Si hay archivos antiguos renombrados o copiados, se incluyen incorrectamente.

### llm-routing
`llm_call.py` · ALTA · `_call_deepseek` no maneja `urllib.error.URLError` (timeout de conexión) ni `json.JSONDecodeError` si la API responde HTML/truncado; la excepción genérica `Exception` lo captura, pero `_fmt_err` para `HTTPError` llama a `e.read()` que puede fallar si el objeto ya fue leído, lanzando `AttributeError` no capturado. Importa porque un timeout o respuesta malformada puede crashear el cron sin log ni notificación.

`llm_call.py` · MEDIA · `_notify_once` escribe `FAIL_LOG` con `run_logger._utc_now()` (módulo externo no mostrado); si `run_logger` no tiene ese atributo o falla, lanza `AttributeError` no capturado, interrumpiendo la notificación. Importa porque silencia fallos críticos del router.

`llm_call.py` · BAJA · `_call_ollama` y `_call_claude` no tienen timeout en `subprocess.run` más allá del parámetro `timeout`; si el contenedor Docker cuelga, el proceso hijo queda huérfano hasta que el timeout del cron lo mate. Importa porque puede acumular procesos zombies en el VPS.

### memory
`memory_sqlite.py` · ALTA · `_fts_query()` no escapa comillas dobles dentro de tokens · Un token como `"M1` genera `"""M1"""` que rompe el MATCH FTS5 y puede causar `sqlite3.OperationalError` silenciado en `recall()`.

`memory_sqlite.py` · MEDIA · `store()` usa `metadata LIKE` para deduplicar por hash · `LIKE` con `%` alrededor del hash JSON es frágil: falso positivo si otro campo contiene el hash, falso negativo si el orden de keys cambia. Debería extraer hash con `json_extract(metadata, '$.hash')`.

`memory_lib.py` · MEDIA · `_mirror_to_db()` captura `Exception` genérica en dual-write · Oculta errores reales (DB corrupta, schema mismatch) que deberían loguearse al menos a stderr.

`memory_lib.py` · BAJA · `now_iso()` usa `datetime.utcnow()` naive (sin timezone) · Inconsistente con `memory_sqlite.now_iso()` que sí usa `timezone.utc`. Causa desfases en timestamps entre DB y .md.

`memory_backfill.py` · BAJA · `parse_table()` filtra líneas que empiezan con `|` pero no líneas vacías intermedias · Una tabla con filas vacías intercaladas se parsea como múltiples tablas, perdiendo filas.

### vault-maintenance
`clean_trash.py` · ALTA · `TRASH_DIRS` usa paths hardcodeados `/home/agent/` en vez de `Path.home()`. Si el VPS cambia de usuario o se ejecuta desde otro entorno, borrará directorios equivocados o fallará silenciosamente.

### autonomous-loops
`learnings.py` · ALTA · `deepseek_propose` recibe `env` y lo pasa a `llm_call.llm_call`, pero `llm_call` no acepta parámetro `env` (firma no mostrada, pero el resto del cluster nunca lo pasa). Si `llm_call` lo ignora silenciosamente, la función usa `os.environ` por defecto, que no contiene las variables del `.env` (cargadas localmente en `learnings.py`). Esto rompe la autenticación si la API key está solo en `.env`.
